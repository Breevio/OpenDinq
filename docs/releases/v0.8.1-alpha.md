# OpenDinq v0.8.1-alpha

v0.8.1-alpha focuses on CI-backed Postgres runtime verification.

This release does not add product features.

## Purpose

The v0.8.0-alpha product flow works locally with MemoryStore and has a PrismaStore code path, but the largest remaining runtime uncertainty was real Postgres verification.

v0.8.1-alpha adds GitHub Actions coverage that runs PrismaStore verification against a Postgres service container.

## What Changed

- Added `.github/workflows/ci.yml`.
- Added a Postgres service container for CI.
- Set CI `DATABASE_URL` to:

```text
postgresql://opendinq:opendinq@localhost:5432/opendinq
```

- CI now runs Prisma Client generation.
- CI applies migrations with `pnpm db:migrate`.
- CI runs `pnpm verify:db` against Postgres.
- CI then runs the full workspace check and high-severity audit.
- DB runtime docs and release checklist now mention CI Postgres verification.

## CI Commands

```bash
pnpm install --frozen-lockfile
pnpm db:generate
pnpm db:migrate
pnpm verify:db
./scripts/check.sh
pnpm audit --audit-level high
```

## Local Validation

Run locally without Postgres:

```bash
pnpm typecheck
pnpm test
pnpm build
./scripts/check.sh
pnpm audit --audit-level high
pnpm db:generate
pnpm db:validate
git diff --check
```

Run locally with Postgres:

```bash
docker compose up -d postgres
DATABASE_URL="postgresql://opendinq:opendinq@localhost:5432/opendinq" pnpm db:migrate
DATABASE_URL="postgresql://opendinq:opendinq@localhost:5432/opendinq" pnpm verify:db
```

## Remaining Limitations

- No product behavior changes in this release.
- Local Postgres verification still depends on Docker/Postgres availability.
- CI verifies PrismaStore runtime behavior, but production deployment hardening is still future work.
- Auth, ownership, billing, teams, LinkedIn/X scraping, browser automation, and production semantic/vector search remain out of scope.

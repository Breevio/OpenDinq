# OpenDinq v0.8-alpha

v0.8-alpha turns the v0.7 technical baseline into a more complete product journey.

```text
Generate Profile -> Workspace -> Cards -> Public Profile -> Discover -> MCP/API automation
```

## What Changed

- Added a product homepage with Generate and Discover entrypoints.
- Added a local profile workspace at `/u/:handle/workspace`.
- Added readiness scoring and next-step guidance.
- Added claim review APIs and UI.
- Added card editor lite for title/content/visibility/order.
- Added deterministic card regeneration.
- Added local-alpha profile draft/published state.
- Improved generation completion to link users to workspace, public profile, and Discover.
- Reused evidence drawers across public profiles, workspace, and Discover.
- Expanded MCP tools for workspace, claim, card, regenerate, and publish actions.
- Extended DB verification to cover v0.8 workspace behavior.

## New / Changed APIs

```text
GET   /api/people/:handle/workspace
GET   /api/people/:handle/claims
PATCH /api/claims/:claimId
POST  /api/cards/:cardId/regenerate
PATCH /api/people/:handle/publish
```

Existing card APIs remain:

```text
GET   /api/people/:handle/cards
PATCH /api/cards/:cardId
POST  /api/people/:handle/cards/manual-note
```

## DB Runtime Status

`pnpm verify:db` now verifies:

- Prisma connection
- profile generation
- workspace summary
- claim approval/rejection
- card edit/regenerate/visibility/order
- manual note cards
- profile publish status
- hidden-card public filtering
- DB-backed search
- evidence retrieval
- Prisma reconnect persistence

Run locally:

```bash
docker compose up -d postgres
pnpm db:migrate
DATABASE_URL="postgresql://opendinq:opendinq@localhost:5432/opendinq" pnpm verify:db
```

## Known Limitations

- Not production-ready.
- No production auth, ownership, or claim verification.
- Workspace is local-alpha behavior and not a security boundary.
- Publishing is draft/published product state, not a permissions system.
- Semantic/vector search is not production runtime.
- Card regeneration is deterministic and evidence-bound.
- No full rich text card editor.
- Connector quality depends on source data.
- No LinkedIn/X scraping, browser automation, or private DINQ APIs.

## Next Milestone

The next useful milestone is CI-backed Postgres verification plus a small amount of workflow polish. Avoid adding new connectors until workspace, publishing, and DB verification are stable.

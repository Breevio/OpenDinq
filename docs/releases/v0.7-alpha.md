# OpenDinq v0.7-alpha

OpenDinq v0.7-alpha is a product alpha for evidence-backed AI-native profile generation and people discovery.

Primary flow:

```text
Generate Profile -> Cards -> Public Profile -> Discover -> MCP/API automation
```

## What Works

- Multi-source profile generation
- Evidence-backed claims
- Card-first public profiles
- Card ordering and visibility
- Manual note cards
- Card patch API
- Evidence drawers on public profile cards
- Discover search across people, claims, cards, and artifacts
- Matched claims, cards, artifacts, top skills, evidence, and profile URL in search results
- API-backed MCP tools
- MemoryStore default runtime
- PrismaStore/Postgres runtime path

## New APIs

- `POST /api/profiles/generate`
- `GET /api/profile-runs/:runId`
- `GET /api/people/:handle/cards`
- `PATCH /api/cards/:cardId`
- `POST /api/people/:handle/cards/manual-note`

Compatibility APIs still available:

- `POST /api/import/github`
- `GET /api/cards/:handle`
- `POST /api/cards/:handle/note`

## Pages

- `/generate`
- `/discover`
- `/u/:handle`
- `/import` as a legacy GitHub-oriented entrypoint

## MCP Updates

The MCP server remains API-first and includes:

- `opendinq_generate_profile`
- `opendinq_get_profile_run`
- `opendinq_search_people`
- `opendinq_get_profile`
- `opendinq_get_evidence`
- `opendinq_create_note_card`
- `opendinq_import_github_profile`
- `opendinq_list_cards`

`opendinq_create_note_card` uses the public manual-note card API.

## Card System Improvements

- Stable card ids
- Stable card ordering
- `public`, `private`, and `hidden` visibility states
- Public filtering for hidden cards
- Manual note cards with manual evidence
- Patch API for title, content, visibility, and order

## Discover Improvements

Search results include:

- person
- score
- explanation
- evidence
- matchedClaims
- matchedCards
- matchedArtifacts
- topSkills
- profileUrl

Search is currently rule/full-text hybrid. It is not production semantic/vector search.

## DB Runtime Status

PrismaStore supports the same API-facing behavior as MemoryStore for profile generation runs, profile sources, people, artifacts, claims, cards, card visibility/order, manual note cards, card patching, search, and evidence retrieval.

Added:

```bash
DATABASE_URL=... pnpm verify:db
```

The verifier checks Prisma connection, profile generation, manual note cards, card patching, hidden-card filtering, DB-backed search, evidence retrieval, and profile/search persistence after Prisma reconnect.

In the current authoring environment, Docker daemon was unavailable, so full local Postgres E2E verification is pending local Docker/Postgres availability.

## Validation Commands

Run before release:

```bash
./scripts/check.sh
pnpm audit --audit-level high
pnpm db:generate
pnpm db:validate
DATABASE_URL=... pnpm verify:db
pnpm screenshots
```

The `verify:db` command requires a reachable Postgres database.

## Known Limitations

- Not production-ready
- No auth
- No ownership or claim verification
- No teams, billing, credits, or permissions model
- No production semantic/vector search
- Multi-source generation quality depends on source data
- Cards do not yet have a full editor, regenerate workflow, or publishing permissions
- Postgres E2E verification requires local Docker/Postgres
- No LinkedIn/X scraping, private DINQ APIs, browser automation, or login-gated scraping

## Next Recommended Milestone

v0.8 should focus on productionizing the persisted runtime:

- run `pnpm verify:db` in CI with Postgres
- add DB migration/release policy
- add integration tests against a real Postgres service
- add card edit/regenerate basics behind existing card APIs
- improve Discover evaluation fixtures without making vector search mandatory
